# 00 — 正式介绍：HVL-Git 专家式探索系统

## 1. 一句话定位

**HVL-Git 专家式探索系统**是一套给 AI 编程工具、研究型项目和复杂系统研发使用的工作协议。它把复杂目标拆成可验证的假设，用 Git 记录可回退状态，用 `.agent/*` 文件记录推理和实验过程，让 AI 不再只是“连续尝试”，而是像优秀专家一样进行有证据、有回退、有复盘的探索。

它尤其适合这类问题：

> 没有现成标准答案，但可以通过实验、日志、训练曲线、仿真、测试或人工验收一步步逼近答案的问题。

## 2. 它不是什么

它不是万能 prompt，不是项目管理软件，也不是让 AI “自动完成所有研究”的魔法。

它真正提供的是一个外部化的探索秩序：

```text
目标
 → 决策节点
 → 假设/子假设
 → Git 分支或 checkpoint
 → 最小实验
 → 验证证据
 → 失败分类
 → 回退/重试/换方案/继续
 → 更新持久化记录
```

如果问题本身没有反馈信号、没有验证标准、没有可观察证据，这套系统也无法凭空判断对错。

## 3. 这套系统适合解决什么样的问题

最适合的问题通常有 8 个特点：

1. **没有唯一标准答案**：不是查文档就能完成，世界上也可能没有成熟方案。
2. **需要多轮实验**：一次实现无法证明方向正确，需要不断试、测、改。
3. **假设很多且相互耦合**：一个大方案下面有多个子假设，比如算法、数据、仿真、工程实现、评估指标都可能出问题。
4. **失败原因不明显**：结果不好时，可能是方案错、执行错、验证错、环境错、数据错，也可能只是随机性导致。
5. **反馈有噪声或有延迟**：例如强化学习训练、仿真实验、大模型评测、复杂系统压测。
6. **改动风险较高**：一次错误的大改可能污染代码、环境或实验结论。
7. **需要长期追溯**：几天后、几周后，甚至换人/换 AI 后，还需要知道之前为什么这么做。
8. **失败本身有价值**：失败实验能缩小搜索空间，告诉团队哪些路已经被证伪。

简单说：

> 只要一个任务更像“研究/探索/排障/系统优化”，而不是“照说明书实现”，就值得考虑使用 HVL-Git。

## 4. 典型适用场景

### 4.1 科学研究与工程研究

适合做计算实验、算法研究、系统研究、实验设计、指标对比和消融实验。

这类任务的关键不是“直接写出最终答案”，而是持续回答：

- 当前假设是什么？
- 证据是否支持它？
- 如果失败，失败在哪一层？
- 下一个更小、更可验证的实验是什么？

### 4.2 强化学习训练与机器人/具身智能项目

非常适合强化学习、仿真训练、机器人控制、自动插取/叉取、导航、抓取、装配等任务。

以“强化学习训练自动插取/叉取能力”为例，失败可能来自很多层：

- 奖励函数设计不合理；
- 状态观测缺少关键信息；
- 动作空间过粗或过细；
- 仿真环境和真实场景差距太大；
- 训练 seed 或超参数导致偶然结果；
- 成功率指标定义不完整；
- 轨迹看似成功，但存在碰撞、卡顿、不可迁移等隐藏问题。

这种项目最怕 AI 或工程师只说“再训一次”“再调一下参数”。HVL-Git 要求每次训练前明确假设，每次训练后记录证据，并判断是否应该回到奖励函数、环境建模、策略结构、评估指标或工程实现层。

### 4.3 元学习、AutoML 与没有标准答案的算法探索

元学习项目也非常适合。

原因是元学习往往不是单一代码 bug，而是多个层级同时影响结果：

- 任务分布是否合理；
- inner loop / outer loop 是否匹配；
- few-shot 设置是否稳定；
- 训练任务和测试任务是否泄漏或错配；
- 表征是否可迁移；
- baseline 是否公平；
- 评估是否真正衡量了“快速适应能力”。

这类问题没有一个全球统一答案，只能通过假设、实验、对照、消融和失败复盘逼近。

### 4.4 超复杂软件系统与 AIoT 系统

适合用于：

- 多模块系统架构重构；
- 视频分析、多模态感知、边缘计算、设备接入、云边协同；
- 分布式系统、数据管线、实时推理服务；
- 性能优化、稳定性治理、安全生产类系统；
- 多团队、多 Agent 接力研发。

这些系统的难点通常不是某一行代码，而是链路长、边界多、现象和根因距离远。HVL-Git 可以把“现象—假设—验证—根因—修复”变成可追踪链路。

### 4.5 复杂 Debug 和根因分析

适合用于那种“看起来哪里都可能有问题”的 bug：

- 偶现问题；
- 性能退化；
- 训练结果突然变差；
- 某个模型、服务或设备只在特定场景失败；
- 日志、数据、环境、代码多方耦合。

这里 Git 的价值尤其大：可以用 checkpoint、branch、diff、bisect、restore 等机制定位引入问题的具体变化。

## 5. 不适用或不建议使用的场景

HVL-Git 不适合所有任务。

不建议用于：

1. **简单确定性任务**：例如一两个字段改名、补一个小配置、改一个明显 typo。
2. **已有成熟做法且风险低的任务**：直接照最佳实践实现即可，不必建立完整假设树。
3. **完全没有验证手段的任务**：没有测试、日志、指标、人工验收标准，系统无法判断假设是否成立。
4. **纯主观偏好的任务**：例如“哪个颜色更好看”，除非能转化为用户测试或明确评审标准。
5. **必须一次性快速处理的低价值任务**：流程成本可能高于收益。
6. **不能保存过程记录的敏感任务**：如果实验记录、代码 diff、日志里包含秘密、隐私或受限数据，必须先建立脱敏与权限边界。

一句话：

> HVL-Git 不是为了把简单事情复杂化，而是为了让复杂探索不失控。

## 6. 它和 Git、实验追踪工具的关系

HVL-Git 不替代 Git，也不替代 MLflow、Weights & Biases、TensorBoard、Jupyter Notebook 或内部实验平台。

更合理的分工是：

```text
Git
  记录代码状态、分支、commit、diff、可回退点。

.agent/*
  记录为什么这样做、假设是什么、失败如何解释、下一步怎么决策。

实验追踪工具
  记录训练 run、参数、seed、指标、曲线、模型 artifact、视频、评估结果。

测试/日志/仿真/人工验收
  提供证据，判断假设是否成立。
```

对于机器学习和强化学习项目，推荐原则是：

> **Git 按“概念假设”建分支；实验平台按“训练 run”记录结果。**

不要为每一个 seed 或每一次超参数微调都创建 Git 分支。Git 分支应该对应“奖励函数改法”“观测空间改法”“策略结构改法”“评估指标改法”这类概念变化。具体训练次数、seed、曲线和 artifact 交给实验追踪工具记录。

## 7. 对强化学习自动插取/叉取项目的建议用法

建议把每个关键问题拆成决策节点：

```text
N0：目标能力是否定义清楚？
N1：成功率指标是否可靠？
N2：奖励函数是否能引导插取动作？
N3：观测空间是否包含对准所需信息？
N4：动作空间是否适合稳定控制？
N5：仿真环境是否覆盖真实失败场景？
N6：策略是否能跨物体、姿态、光照、误差泛化？
N7：sim-to-real 迁移是否需要额外机制？
```

每个节点下面再放假设。例如 N2 奖励函数节点：

```text
H2-A：当前奖励太稀疏，导致策略无法学习对准过程。
H2-B：距离奖励权重过高，导致策略追求接近但不完成插入。
H2-C：缺少碰撞惩罚，导致训练中出现不可用动作。
H2-D：成功奖励定义太宽，导致策略学到投机行为。
```

每次实验至少记录：

- Git branch / commit；
- 环境版本；
- reward 配置；
- observation / action space；
- 算法与超参数；
- seed；
- 训练步数；
- 成功率、碰撞率、平均完成时间、失败模式；
- 代表性轨迹视频或日志；
- 是否支持当前假设；
- 下一步退回哪个节点或进入哪个节点。

## 8. 对元学习项目的建议用法

建议先把元学习项目拆成以下节点：

```text
N0：元学习目标到底是什么？快速适应、少样本泛化、跨域迁移，还是自动调参？
N1：任务分布是否定义正确？
N2：baseline 是否公平且足够强？
N3：inner loop 设计是否稳定？
N4：outer loop 优化目标是否匹配最终评估？
N5：评估协议是否存在泄漏或偏差？
N6：模型表征是否真的可迁移？
N7：消融实验是否能解释增益来源？
```

元学习最容易出现“看起来提升了，但不知道为什么提升”的问题。HVL-Git 要求把每个提升拆成可验证假设：是任务采样变了、模型容量变了、优化过程变了，还是评估方式变了。

## 9. 什么时候必须启用这套系统

建议团队设置一个硬触发条件。满足任意一条，就启用 HVL-Git：

- 预计任务超过半天，且结果不确定；
- 有两个以上可行方案需要比较；
- 失败后无法一眼看出原因；
- 涉及训练、仿真、benchmark、性能优化或复杂 debug；
- 需要多人或多个 AI 接力；
- 任务会产生大量实验记录；
- 失败经验本身值得保留；
- 代码改动一旦混杂就很难回退。

## 10. 什么时候只用轻量版

对中等复杂任务，可以只保留最小闭环：

```text
.agent/current-plan.md
.agent/experiment-log.md
Git checkpoint
验证命令
```

不必强行维护完整决策树。

## 11. 判断成功的标准

使用 HVL-Git 后，一个复杂任务是否“做得好”，不只看最终有没有跑通，还看：

1. 目标和成功标准是否明确；
2. 每个关键方案是否被表达成假设；
3. 每次尝试是否有验证证据；
4. 失败是否被分类，而不是被遗忘；
5. Git 是否能回到关键节点；
6. 后来的人或 AI 是否能看懂之前为什么这么做；
7. 失败实验是否减少了后续搜索空间。

## 12. 这个系统本质上属于哪类问题

这个系统不是一个“全新科学方法”，而是一个老问题在 AI 编程时代的新形态。

它组合了几个成熟母问题：

- 科学研究中的假设验证；
- 工程调试中的最小复现与证据定位；
- 机器学习中的实验追踪与消融实验；
- 软件工程中的版本控制与回滚；
- 复杂系统治理中的状态外部化；
- 专家工作法中的复盘和决策记录。

真正的新点在于：

> 把这些机制包装成 AI 编程工具能稳定遵守的规则、文件结构、Git 工作流和可交接项目包。

## 13. 推荐读者

这份项目包适合给以下人看：

- AI 编程工具重度用户；
- 算法工程师、机器人研发、强化学习研发；
- 科研型工程团队；
- 做 AIoT、视频分析、工业智能化、复杂系统集成的团队；
- 希望让 Codex、Cursor、Claude Code 等工具承担更长期、更复杂任务的人；
- 需要把复杂探索过程交接给他人或下一个 AI 的团队。

## 14. 外部参考

这些工具和文档不是 HVL-Git 的前提，但它们说明了本系统为什么要这样设计：

- OpenAI Codex 支持用 `AGENTS.md` 为项目提供持久化指令，并会在开始工作前读取相关指导。
- OpenAI Codex Skills 使用 `SKILL.md` 描述可复用工作流和任务触发范围。
- Cursor 建议把项目规则保持聚焦，并把规则提交进 Git 让团队共享。
- Git 的 `worktree`、`restore`、branch、commit 等能力可以作为探索过程的状态与回退机制。
- MLflow 等实验平台提供机器学习和 Agent 工作流的实验追踪、评估和优化能力。
