# 09 — 本项目包来源：对话内容整理

这份项目包来自一次关于“如何让 AI 编程工具像优秀专家一样探索复杂问题”的讨论。

## 初始想法

当 AI 解决复杂问题时，不应该只是生成一个方案并一直往下写。

更合理的方式是：

1. 先制定计划；
2. 把计划中的关键判断拆成假设；
3. 对每个假设做尝试验证；
4. 如果方案 A 不行，要回到当前决策节点；
5. 判断是方案 A 执行错了，还是方案 A 本身不成立；
6. 如果当前节点解决了，可以进入下一个节点；
7. 如果当前节点不成立，可能要退回上一个节点；
8. 过程中必须持久化记录假设、结果、反思和决策。

## 为什么要持久化

如果只依赖聊天上下文，AI 容易因为上下文压缩、任务中断或模型切换而失控。

所以需要把关键状态保存到文件里，例如：

- 当前计划；
- 决策树；
- 假设列表；
- 实验记录；
- 验证结果；
- 失败反思；
- 下一步动作。

## 后续关键补充

进一步判断：要做好这件事，必须把 Git 用好。

Git 不只是保存代码，而是 AI 探索过程中的外部状态机：

- 每个关键决策点有 checkpoint；
- 每个方案有独立分支或 commit；
- 每次实验有验证结果；
- 失败分支不一定删除，而是作为证据保留；
- AI 可以通过 Git diff/log/status 知道自己做了什么、怎么回去。

## 最终形成的核心协议

```text
目标
 → 决策节点
 → 假设
 → Git 分支/commit
 → 最小实验
 → 验证结果
 → 失败分类
 → 回退/重试/换方案/继续
 → 更新 .agent 状态文件
```

## 新增关键补充：适用范围

进一步讨论后，明确了一个很关键的问题：这套系统到底适合解决什么问题？

新的判断是：HVL-Git 不是给所有任务用的，而是特别适合“没有标准答案，但可以通过实验验证逐步逼近”的问题。

典型场景包括：

- 科学研究；
- 超复杂系统；
- 强化学习训练自动插取/叉取；
- 元学习项目；
- 机器人、仿真、具身智能；
- 多方案算法探索；
- 复杂系统 Debug 和根因分析。

尤其是在强化学习和元学习项目里，世界上往往没有一个统一标准答案。真正能做的是：提出假设，设计实验，验证结果，失败后判断是方案错、实现错、验证错、环境错，还是需要退回上一个决策节点。

因此，本项目包新增了：

- `docs/00-formal-introduction.md`：正式介绍和适用范围；
- `docs/11-rl-meta-research-guide.md`：强化学习、元学习和科研型项目指南；
- `.agent/project-fit.md`：项目适配性判断模板；
- `templates/research-task-brief.md`：科研/探索任务简报模板；
- `templates/ml-rl-experiment-entry.md`：机器学习/强化学习实验记录模板；
- `examples/rl-auto-insertion/README.md`：强化学习自动插取/叉取示例；
- `examples/meta-learning/README.md`：元学习示例。
