# 02 — 核心模型

## 一句话模型

```text
目标 → 决策节点 → 假设 → 最小实验 → 验证证据 → 失败分类 → 回退/继续/换方案
```

## 三个系统

### 1. 推理状态系统：`.agent/*`

`.agent/*` 保存“为什么”。

它记录：

- 当前计划；
- 决策树；
- 假设和子假设；
- 实验记录；
- 验证方式；
- 风险；
- 交接摘要。

### 2. 代码状态系统：Git

Git 保存“做了什么”。

它记录：

- 改了哪些文件；
- 什么时候改的；
- 当前在什么分支；
- 哪个点可以回退；
- 哪些方案已经尝试过。

### 3. 证据系统：测试/日志/运行结果

验证系统保存“是否成立”。

它可以是：

- 自动化测试；
- 类型检查；
- lint；
- 构建结果；
- 运行日志；
- benchmark；
- 人工验收步骤；
- 生成物检查。

## 为什么三者缺一不可

只用 `.agent/*`：会知道为什么，但代码状态不可回退。

只用 Git：会知道改了什么，但不知道为什么这样改。

只用测试：会知道结果对不对，但不知道失败后该回到哪个决策节点。

三者合起来，AI 才有稳定探索能力。

## 决策树 vs 线性计划

复杂任务不能只用线性计划：

```text
Step 1 → Step 2 → Step 3 → Step 4
```

因为失败后通常不是继续 Step 5，而是要判断：

- Step 3 做错了吗？
- Step 2 的假设错了吗？
- Step 1 的问题定义错了吗？
- 是不是缺一个前置条件？
- 是不是验证方法本身不对？

所以复杂任务应该是决策树：

```text
Node 1
 ├─ Hypothesis A → fail → classify → retry / abandon / split
 ├─ Hypothesis B → pass → Node 2
 └─ Hypothesis C → blocked → resolve prerequisite
```

## 最重要的规则

失败后不要继续堆补丁。先回答一个问题：

> 当前失败，是实现错了，还是假设错了？

这个判断如果不做，后面很容易变成“AI 在泥坑里刨地”。
